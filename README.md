## 技術選定

本アプリケーションでは、以下の技術を採用しました。

| 技術                      | 用途                  | 選定理由                                                                                              |
| ------------------------- | --------------------- | ----------------------------------------------------------------------------------------------------- |
| TypeScript                | アプリケーション全体  | 業務で利用経験があり、型安全性によって実装時の不具合を早期に検出できるため                            |
| Hono                      | Web APIフレームワーク | 業務でNestJSとHonoの利用経験があり、今回は小規模なAPIのため、シンプルな構成で実装できるHonoを採用した |
| Prisma                    | ORM                   | 業務で利用経験があり、型安全なデータベースアクセスとマイグレーション管理を一元化できるため            |
| PostgreSQL                | データベース          | 業務で利用経験があり、リレーショナルデータのモデリングやインデックス設計などの知見を活かせるため      |
| Zod / `@hono/zod-openapi` | 入力値検証・API仕様   | Honoとの親和性が高く、入力値検証とOpenAPI定義を一元管理できるため                                     |
| Vitest                    | テスト                | 業務でJestとVitestの利用経験があり、実行速度が速く、TypeScriptとの親和性も高いためVitestを採用した    |

## アーキテクチャ

本アプリケーションでは、データを更新する処理と参照する処理を分離するため、CommandとQueryを分けた構成を採用しています。

```text
.
├── command
│   ├── controller
│   ├── domain
│   ├── infra
│   └── usecase
└── query
    ├── controller
    ├── infra
    ├── todo
    └── usecase
```

- Command側には、TODOやユーザーの作成・更新・削除など、状態を変更する処理を配置しています。
- Query側には、TODO一覧取得など、データを参照する処理を配置しています。
- データベースはCommandとQueryで同じPostgreSQLを使用しています。
  - ただし、アプリケーションコード上では更新処理と参照処理の責務を分離しています。

## 工夫した点

### CommandとQueryの責務を分離した

更新処理と参照処理では求められるモデルが異なるため、それぞれをCommandとQueryに分離しました。

分離した理由は以下のとおりです。

- `updatedAt`は一覧表示には必要ですが、今回のドメインルールでは使用しないため、参照都合だけでEntityに保持させたくなかったため
- 参照処理をRepositoryに持たせると、`findTodosWithUserName`のような画面・API専用のメソッドが増えると考えました。
  - その結果、Repositoryの責務が肥大化するため、参照処理はQuery Serviceに分離しました。

### レイヤーごとに責務を分離した

CommandとQueryの内部では、Controller、Use Case、Domain、Infrastructureの責務を分けています。

| レイヤー        | 主な配置先                               | 責務                                                                   |
| --------------- | ---------------------------------------- | ---------------------------------------------------------------------- |
| Controller      | `command/controller`、`query/controller` | HTTPリクエストの受付、入力値検証、HTTPレスポンスへの変換               |
| Use Case        | `command/usecase`、`query/usecase`       | アプリケーション固有の処理の流れを定義                                 |
| Domain          | `command/domain`                         | Entity、Value Object、ドメインルール、Repositoryインターフェースの定義 |
| Infrastructure  | `command/infra`、`query/infra`           | Prismaを使用したデータベースアクセス                                   |
| Query Interface | `query/todo`                             | Query ServiceのインターフェースとRead Modelの定義                      |

- ControllerからPrismaを直接呼び出さず、Use Caseを経由することで、
  HTTP処理とアプリケーション処理を分離しています。
  - これにより、入力形式やレスポンス形式の変更と、TODOの作成・更新処理の変更をそれぞれ独立して扱いやすくしています。
- アプリケーションサービスとして集約する構成も考えました。
  - 機能追加による責務の肥大化を避けるため、ユースケース単位でクラスを分割しました。
- Command側のUse CaseはRepositoryインターフェースに依存し、
  Prismaの具体的な実装はInfrastructure側に配置しています。
  - これにより、アプリケーション層を永続化の実装から分離しています。
  - ORMやデータアクセス方法を変更した場合の影響も、局所化できるようにしています。

### HTTPの入力値検証とドメインルールを分けた

Controller側のスキーマでは、リクエスト形式や型、必須項目などを検証します。

Domain側では、タイトルの文字数や優先度の範囲など、ビジネス上の制約を検証します。

| 検証場所   | 検証内容                                           |
| ---------- | -------------------------------------------------- |
| Controller | 必須項目、データ型、文字列形式など                 |
| Domain     | タイトルの文字数、優先度の範囲、所有者変更禁止など |

入力値検証とドメインルールを分けることで、API境界では分かりやすいエラーを返しつつ、Domain層でも不正な状態を防いでいます。

### カーソルページネーション

課題要件である「TODOの数が大量に登録されてもレスポンスが遅延しないような負荷対策」を考慮しました。

一覧取得には、OFFSETではなくIDを基準としたカーソルページネーションを採用しました。

```ts
const todos = await prisma.todo.findMany({
  take: limit + 1,
  ...(cursor
    ? {
        cursor: { id: cursor },
        skip: 1,
      }
    : {}),
  orderBy: {
    id: 'asc',
  },
});
```

- OFFSET方式では、後ろのページになるほど読み飛ばすレコード数が増えてしまい、無駄が生じます。
- 一方、カーソルページネーションでは、前回取得した位置を基準に次のデータを検索できます。
  - 必要最低限のレコード取得で済むため、OFFSET方式で起きる、後半のページになるにつれてレスポンスが遅くなる問題を防げます。
- カーソルページネーションは、ページ番号を指定して任意のページへアクセスするUIとは相性が良くありません。
  - 今回は、課題要件である大量データへの負荷対策を優先して採用しました。

### クエリに合わせたインデックスを設定した

カーソルページネーションと同様に負荷対策でインデックスを設定しました。

TODOをユーザーごとに取得し、ID順でページネーションすることを想定し、`userId`と`id`の複合インデックスを設定しています。

```prisma
@@index([userId, id])
```

## テスト方針

- Todo ドメインで定義している Entity と Value Object は、外部サービスやデータベースに依存しません。
  - ドメインルールを保持しているため、単体テストで確認しています。
- APIについては、各レイヤーを個別にモックしてテストするのではなく、HTTPリクエストからデータベースへの反映までを確認することを重視しました。
  - 各レイヤーが一連の流れとして連携できているかや 取得結果やエラー応答を結合テストで検証しています。

テスト対象は以下のとおりです。

| 対象                      | テスト種別 | 主な確認内容                                                            |
| ------------------------- | ---------- | ----------------------------------------------------------------------- |
| Todo Entity・Value Object | 単体テスト | ドメインルール、境界値、不正な状態の防止                                |
| Command API               | 結合テスト | Todo/User の作成・更新・削除、存在確認、エラー応答、PostgreSQL への反映 |
| Query API                 | 結合テスト | Todo の一覧取得、検索条件、取得順序、カーソルページネーション           |

GitHub Actionsでは、PR作成・更新時にCI（`.github/workflows/ci.yml`）を実行しています。

Lint、テスト、ビルドを実行することで、既存の処理が壊れていないことを継続的に確認できるようにしています。

## 気づき・学び

### テストコードの価値を実感した

エラークラスのリファクタリングを行った際、テストコードを実行しました。  
その結果、既存の挙動が維持されていることをすぐに確認できました。  
もしテストコードがなければ、毎回APIを起動してリクエストを送信し、動作確認を行う必要があります。  
今回の課題を通して、リファクタリングを安心して進められるという点で、テストコードの価値を改めて実感しました。

### 初期構築の難しさ

普段の業務では、既存プロジェクトに対してAIを活用しながら開発することが多く、既存の設計や実装方針に沿って進められます。  
一方、今回はゼロからアーキテクチャやディレクトリ構成、テスト方針などを決める必要がありました。  
設計の方向性を考える時間は想像以上にかかり、初期構築では実装以上に設計の判断が求められることを実感しました。

### 本で学んだ知識を実装で試せた

カーソルページネーションは書籍で学んだことはありましたが、実際に実装したのは今回が初めてでした。  
過去の業務では、OFFSETによるページネーションを実装した後にカーソルページネーションを知りました。  
カーソル方式の方が良いかもしれないと感じた経験があったため、今回の課題では実際に採用しました。  
実装を通して、データ件数が増えても安定した性能を期待できることを確認しました。  
一方で、ページ番号を指定して任意のページへ移動するUIとは相性が良くないことも理解できました。  
書籍で得た知識を実装で検証し、メリットだけでなくトレードオフまで体験できたことは大きな学びになりました。

## 生成AIの利用について

### 使用したツール

- Codex（GPT-5.6 Luna Xhigh）
- Cursor Tab

### 利用方針

生成AIは設計、実装、レビューを支援する開発ツールとして利用しました。

### リポジトリルール・スキル

生成AIが既存の設計方針に沿って実装できるよう、
リポジトリ直下に `AGENTS.md` を配置し、実装ルールを定義しました。

また、繰り返し行う作業はスキルとして定義しています。

- `.codex/skills/feature-implementation/SKILL.md`
  - 要件確認、既存コードの調査、実装、検証、レビューまでの実装フロー

- `.codex/skills/testing/SKILL.md`
  - テスト方針、テストケース作成、テスト実行、型チェック、Lintまでのテストフロー

これらのルール・スキルを参照させることで、実装方針やテスト方針を統一しました。

その際、スキルで定義したルールに沿ってAIにテストコードを作成させ、内容をレビューした上で採用しました。

### 主な利用内容

#### 初期実装

- プロジェクトの初期構築や最初のAPI（Create Todo、Update Todo）の実装では、Cursor Tabを活用しました。
- 初期は、既存の設計方針を十分に把握できていない状態でAIに実装を任せると、意図しない実装になる可能性があると考えました。
- そのため、Create Todo、Update TodoまではCursor Tabを活用しつつ、自分で確認しながら実装を進めました。

#### 実装

主に以下のような内容で利用しています。

- Create Todo、Update Todo以外のAPIの実装
- エラークラス等のリファクタリング
- カーソルページネーションの設計・実装の相談
- CommandとQueryの責務分離に関する相談
- READMEのレビュー・文章推敲

#### テスト

テストコードについては、リポジトリ固有のテストスキルを利用し、一貫した方針で実装しました。

また、AIには既存テストと実装を読ませ、

- 境界値
- エラーケース
- ページネーション
- ユーザー分離

などの観点が不足していないかレビューさせ、最終的なテストケースの採用可否は自分で判断しました。

### 利用時に意識したこと

- AIの提案をそのまま採用しない
- 課題要件・既存コード・テスト結果と照らし合わせて判断する
- 型チェック、Lint、テストを実行して変更内容を確認する

最終的な設計判断、実装内容、コードレビュー、テスト結果の確認はすべて自分で行いました。
