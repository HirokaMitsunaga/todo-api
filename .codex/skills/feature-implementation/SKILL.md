# 機能実装

## 概要

このワークフローは、機能追加や仕様変更を行う際に使用します。
要求された範囲に集中し、既存の設計・実装方針を尊重しながら、実装後は十分に検証を行ってから結果を報告してください。

## ワークフロー

1. 要件を読み、明確な内容・曖昧な内容・確認なしに拡張してはいけない内容を整理する。
2. 実装前に、関連するコード・設定・テスト・ドキュメントを確認する。
3. 変更範囲が広い場合や複数ファイルにまたがる場合は、実装前に簡潔な実装方針を共有する。
4. 要件を満たすために必要最小限の変更を実装する。
5. 振る舞いが変わる場合や回帰の可能性がある場合は、必要なテストを追加または更新する。
6. 必要に応じて型チェック、Lint、テスト、ビルドなどの検証を実行する。
7. 差分を確認し、不要な変更・意図しないフォーマット変更・テスト漏れ・考慮不足なケースがないかを確認する。
8. 最後に、変更内容・設計判断の理由・実施した検証内容・実行できなかった検証があればその理由を報告する。

## ガードレール

- 要件に関係のないファイルは変更しない。
- 要件上必要でない限り、新しい依存ライブラリは追加しない。追加する場合は理由を説明する。
- 明確な理由がない限り、既存のアーキテクチャやコーディング規約を変更しない。
- 要件が曖昧な場合でも、仕様を勝手に拡張して実装しない。
- 新しいツールを導入するよりも、既存のスクリプトやプロジェクトで定義されているコマンドを優先して利用する。
