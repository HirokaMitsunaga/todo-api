# Unit Testing

## 基本方針

- TypeScriptで実装する。
- テストフレームワークはVitestを使用する。
- ドメイン層のみ単体テストとする。
- ドメイン層以外は結合テストとする。
- ドメイン層はホワイトボックステストとする。
- 同じ構造のテストは `it.each` によるテーブル駆動テストで実装する。
- テストケース名は日本語で記述する。
- テストケース名の先頭には必ず `【正常系】` または `【異常系】` を付ける。

## 単体テスト

- Entity
- Value Object
- Domain Service
- ドメインルール
- 状態遷移
- ドメインエラー

public APIを通して振る舞いを検証し、privateな実装は直接テストしない。

## 結合テスト

ドメイン層以外は結合テストとして扱う。

依存関係をすべてモックした単体テストは作成しない。

## 優先してテストする内容

優先順位は次のとおり。

1. ドメインルール
2. 条件分岐
3. 状態遷移
4. バリデーション
5. 計算処理
6. エラー

次のような仕様のみの確認は、原則としてテストしない。

- コンストラクタで渡した値をgetterが返すだけ
- 実装をそのまま確認するだけのテスト
- 業務仕様に影響しない実装詳細

## テスト対象の判断

- 単純なgetter・setterなど、実装をそのまま確認するだけのテストは作成しない。
- 条件分岐、計算、バリデーション、状態遷移、ドメインルールがある処理をテストする。
- public APIを通して仕様を検証し、実装詳細には依存しない。
- 参照が同じであることよりも、仕様として保証すべき振る舞いを優先して検証する。
- 同じコードを複数回実行して例外を検証しない。1回の実行で十分な場合は1回だけ実行する。
- カバレッジを上げることだけを目的としたテストは作成しない。

## 実装後

- 対象テストを実行する。
- 全テストを実行する。
- 型チェックを実行する。
- lintを実行する。
- 実施内容と結果を報告する。
